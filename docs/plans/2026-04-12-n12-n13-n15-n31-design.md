## N12: SSN Encryption via Supabase Vault

### Problem
`driver_applications.ssn_encrypted` is always NULL. The full SSN is received, last-4 extracted, then discarded. Carriers need the full SSN for background checks, Clearinghouse queries, and tax reporting.

### Solution
1. **Migration:** Enable `pgcrypto` extension, create a Vault secret (`ssn_encryption_key`), create two `SECURITY DEFINER` RPCs:
   - `encrypt_ssn(p_application_id uuid, p_ssn text)` — encrypts and stores
   - `decrypt_ssn(p_application_id uuid)` — decrypts and returns (admin/safety only)
2. **Server action change:** After extracting `ssn_last4`, call `encrypt_ssn` RPC
3. **Access control:** The decrypt RPC checks `get_tenant_id()` + role via `app_metadata`

### Files
- `supabase/migrations/20260412130000_ssn_encryption.sql` (NEW)
- `src/app/actions/driver-applications.ts` (modify `updateDraftSection` page1 handler)

## N13: Account Deletion (GDPR/CCPA)

### Problem
No way to delete a user account. GDPR Art. 17 and CCPA 1798.105 require it.

### Solution
Server action `deleteMyAccountAction()`:
1. Verify user identity (require password confirmation)
2. Cancel Stripe subscription (if active)
3. Nullify PII on `tenant_memberships` (full_name, email → null)
4. Nullify PII on `tenants` (name, phone, address → anonymized)
5. Mark all drivers owned by tenant as `driver_status = 'inactive'`
6. Ban the Supabase Auth user (prevents re-login)
7. Log audit event (actor = system, action = 'account:deleted')
8. Sign out

Financial records (orders, payments, invoices) are RETAINED with anonymized tenant name for tax/compliance. This is a soft-delete + PII scrub, not a hard cascade delete.

### Files
- `src/app/actions/account.ts` (NEW)
- `src/lib/validations/account.ts` (NEW — Zod schema for delete confirmation)

## N15: Redis Cache Layer (Upstash)

### Problem
Every dashboard load hits the database directly. No general-purpose cache. At 500+ tenants, DB saturates.

### Solution
Thin cache-aside wrapper using existing Upstash Redis:
- `cacheGet<T>(tenantId, key)` — returns cached value or null
- `cacheSet(tenantId, key, value, ttlSeconds)` — stores with TTL
- `cacheInvalidate(tenantId, key)` — deletes cache entry
- `cacheInvalidateAll(tenantId)` — deletes all entries for a tenant

Cache 5 hottest queries:
1. `fetchFinancialSummary` (5 min TTL)
2. `fetchComplianceOverview` (5 min TTL)
3. `fetchExpirationAlerts` (5 min TTL)
4. Dashboard KPI cards (5 min TTL)
5. Order/driver counts for nav badges (2 min TTL)

Invalidation: server actions that mutate orders/compliance/drivers call `cacheInvalidate(tenantId, 'affected-key')` after the DB write.

Fallback: if Redis is unavailable, return null (cache miss) — queries hit DB directly. No hard dependency.

### Files
- `src/lib/cache.ts` (NEW — cache utility)
- `src/lib/queries/financials.ts` (wrap fetchFinancialSummary)
- `src/lib/queries/compliance.ts` (wrap fetchComplianceOverview + fetchExpirationAlerts)

## N31: Critical Path Tests (~40-50 tests)

### Problem
93% of server actions untested. 7 test files for 52 action files.

### Solution
Add tests for the 10 most critical actions using existing mock patterns:
1. `orders.ts` — createOrder, updateOrder, deleteOrder, updateOrderStatus
2. `payments.ts` — createPayment
3. `trips.ts` — createTrip, assignOrderToTrip
4. `drivers.ts` — createDriver, updateDriver
5. `compliance.ts` — createComplianceDoc

Each test covers: Zod validation, authorize() check, tenant isolation, success path, error path.

### Files
- `src/app/actions/__tests__/orders.test.ts` (extend existing)
- `src/app/actions/__tests__/payments.test.ts` (NEW)
- `src/app/actions/__tests__/compliance.test.ts` (NEW)
